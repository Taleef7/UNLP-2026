# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Task Overview

UNLP 2026 shared task: retrieval-augmented QA over Ukrainian PDFs (6-choice MCQ).

**Metric:** `0.5 * answer_acc + 0.25 * doc_acc + 0.25 * page_prox`  
**Submission target:** Kaggle, T4x2, hard wall-clock limit of 9 hours.  
**Best public score:** `v7` at **0.8688** (4th place). Leaderboard leader: 0.9377.

## Commands

Run the full dev benchmark for a preset:
```bash
python3 scripts/benchmark_candidate.py \
  --preset v7_baseline \
  --questions data/dev_questions.csv \
  --output-dir outputs/benchmarks/v7_baseline
```

Run against a named CV split or fold:
```bash
python3 scripts/benchmark_candidate.py \
  --preset v7_baseline \
  --split fold0 \
  --output-dir outputs/benchmarks/v7_baseline_fold0
```

Smoke run (10 questions):
```bash
python3 scripts/benchmark_candidate.py \
  --preset v7_baseline \
  --n-questions 10 \
  --output-dir outputs/benchmarks/smoke
```

Compare two benchmark runs:
```bash
python3 scripts/diff_benchmark_runs.py \
  --base-dir outputs/benchmarks/v7_baseline \
  --candidate-dir outputs/benchmarks/<candidate>
```

Evaluate predictions from a CSV:
```bash
python3 scripts/00_eval_harness.py
```

Regenerate CV and lockbox splits:
```bash
python3 scripts/00_create_splits.py
```

Check parity between shared runtime and rendered Kaggle bundle:
```bash
python3 scripts/check_kaggle_parity.py \
  --preset v7_baseline \
  --n-questions 5 \
  --output-dir outputs/parity/smoke
```

Run tests (use `python3 -m unittest`; `pytest` is not installed):
```bash
# All tests
python3 -m unittest discover -s tests

# Single test module
python3 -m unittest discover -s tests -p "test_benchmark_logic.py"
```

## Architecture

The pipeline has two runtime surfaces that must stay in sync:

1. **Local benchmark** (`scripts/benchmark_candidate.py`) — used for evaluation and candidate vetting
2. **Kaggle submission bundle** (`notebooks/v*_kaggle_submission.py`) — thin wrappers that call `pipeline_shared.py`

Both import from the same core:

- `notebooks/pipeline_shared.py` — all pipeline logic: retrieval, reranking, MCQ answer selection, metric computation, and the `PipelineRunner` class
- `notebooks/pipeline_presets.json` — named presets that configure every pipeline knob; the only difference between candidates is a preset
- `notebooks/kaggle_bundle.py` — renders a standalone Kaggle submission script from `pipeline_shared.py` + a preset

### Pipeline flow (per question)

1. **Retrieval** — BGE-M3 dense-only against precomputed embeddings in `embeddings/`; query is question text only (adding answer options hurts -7.6% Doc@1)
2. **Document rerank** — optional cross-encoder stage over retrieved doc candidates
3. **Page selection** — answer-aware page rerank within the chosen document
4. **MCQ answering** — LLM logprob scoring (MamayLM or LapaLLM); no chat template (chat template hurts MCQ by -0.028)

### Data splits

`data/splits/` contains grouped-by-Doc_ID CV folds and a fixed lockbox split, generated by `scripts/00_create_splits.py`. The primary evaluation protocol uses:
- grouped CV fold 0 + fold 1 (5-fold, but only 2 used for gating)
- lockbox held-out document set

Full-dev results alone are **not** sufficient for promotion.

### Benchmark artifacts

Each benchmark run writes to `outputs/benchmarks/<name>/`:
- `predictions.csv` — one row per question
- `per_question.csv` — detailed per-question scores
- `summary.json` — composite, answer, doc, page scores
- `timings.json` — per-stage timing
- `submission.csv` — Kaggle-format output

## Promotion Gate

A candidate must satisfy **all three** before any Kaggle submission:
1. Full-dev local composite > reproduced `v7_baseline` (0.8635 local)
2. Grouped CV fold 0 + fold 1 both beat `v7_baseline` (0.8308 / 0.8718)
3. Lockbox beats `v7_baseline` (0.8877)
4. Runtime projection < 8.5 hours on T4x2

Current promotion leader: **`v7_baseline`** (no new submissions yet pending a passing candidate).

## Environment Notes

- Python 3.9; `local_packages/` is on `sys.path` at runtime (contains pinned wheels)
- `HF_HOME` defaults to `cache/hf_cache/`
- Benchmark runs should be done on an A30 compute node (`gilbreth-b000`), not the frontend host (causes `llama_cpp` crashes)
- Historical P100 timing notes in older scripts are stale — use T4x2 budgets
